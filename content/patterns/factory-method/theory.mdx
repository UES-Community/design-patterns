## Objetivo

El **Factory Method** define una interfaz para crear objetos en una superclase, pero permite a las subclases alterar el tipo de objetos que se crearán. En lugar de llamar directamente a `new`, el código llama a un método de fábrica.

## Causa

Tienes una clase base que gestiona objetos de un tipo específico, pero necesitas que subclases puedan crear objetos de diferentes tipos concretos sin modificar la clase base.

## Aplicación

1. Declarar el **producto** como interfaz/clase abstracta.
2. La clase **creador** declara el factory method como abstracto o con implementación por defecto.
3. Las **subclases creadoras** sobreescriben el factory method para devolver el tipo concreto apropiado.

## Estructura

```
Creator (abstract)
├── +someOperation(): void
└── +createProduct(): Product  ← factory method

ConcreteCreatorA extends Creator
└── +createProduct(): ConcreteProductA

ConcreteCreatorB extends Creator
└── +createProduct(): ConcreteProductB

Product (interface)
├── ConcreteProductA
└── ConcreteProductB
```

### Cuándo usar

- Cuando no sabes de antemano los tipos exactos de objetos que tu código debe crear.
- Cuando quieres proporcionar a usuarios de tu biblioteca o framework una forma de extender sus componentes internos.
- Cuando quieres ahorrar recursos reutilizando objetos existentes en lugar de reconstruirlos cada vez.









## Patrones relacionados

- **Abstract Factory** — a menudo se implementa con Factory Methods
- **Template Method** — el Factory Method es una especialización del Template Method
- **Prototype** — alternativa a Factory Method cuando no necesitas crear jerarquías de Creator
